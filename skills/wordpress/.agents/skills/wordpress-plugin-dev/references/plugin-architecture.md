## Verify Current Docs First

Verify current plugin header fields, dependency headers, WordPress.org release expectations, block metadata behavior, and script module requirements before release or when targeting a specific WordPress version.

## Single File vs Modular Architecture

Use one `plugin-name.php` when:

- The plugin has one small feature and no build step.
- It registers a shortcode, a simple hook, or a few filters.
- There is no REST surface, no complex admin UI, no custom data model, and no shared services.
- The file remains readable without mixing unrelated concerns.

Use modular architecture when:

- The plugin has admin UI, REST endpoints, blocks, cron, privacy exporters/erasers, custom tables, or multiple integrations.
- The plugin needs tests, Composer autoloading, or reusable services.
- Multiple features need clear ownership boundaries.
- Release packaging must separate source, build assets, templates, and tests.

Rule of thumb: start simple, but split responsibilities before a single file becomes a hidden dependency container.

## Recommended Modern Structure

```text
plugin-slug/
|-- plugin-slug.php
|-- composer.json
|-- package.json
|-- src/
|   |-- Plugin.php
|   |-- Admin/
|   |-- Rest/
|   |-- Blocks/
|   `-- Domain/
|-- includes/
|-- assets/
|   |-- css/
|   `-- js/
|-- templates/
|-- languages/
|-- blocks/
|   `-- example/
|       |-- block.json
|       `-- render.php
|-- build/
|-- tests/
`-- readme.txt
```

- `plugin-slug.php`: bootstrap only. Keep headers, constants, autoload loading, activation/deactivation hooks, and the single call that starts the plugin.
- `src/`: namespaced production classes. Put services, controllers, registrars, and domain logic here.
- `includes/`: procedural compatibility files or small helpers when the project already uses this pattern.
- `assets/`: source or static CSS/JS/images not generated by the build.
- `templates/`: PHP templates for frontend/admin markup. Templates receive prepared data and escape on output.
- `languages/`: translation files when the distribution uses bundled translations.
- `build/`: compiled JS/CSS/assets for release. Commit or package when users will not run a build step.
- `blocks/`: block folders with `block.json`; may contain source or built block metadata depending on project convention.
- `tests/`: PHPUnit, integration, and fixture code. Usually excluded from WordPress.org release packages unless intentionally shipped.

## Namespaces And Composer Autoload

Use Composer PSR-4 autoloading for medium and large plugins:

```json
{
  "autoload": {
    "psr-4": {
      "Vendor\\PluginSlug\\": "src/"
    }
  }
}
```

Bootstrap pattern:

```php
defined( 'ABSPATH' ) || exit;

$autoload = __DIR__ . '/vendor/autoload.php';
if ( file_exists( $autoload ) ) {
	require_once $autoload;
}
```

Guidelines:

- Use one vendor-style namespace such as `Vendor\PluginSlug`.
- Keep global functions, constants, hook names, option names, handles, and meta keys prefixed with the plugin slug.
- Do not rely on Composer dev dependencies in production unless they are packaged intentionally.
- If autoload is required and missing, fail with an admin-only notice rather than a public fatal where possible.

## Plugin Header Fields

Use current official header docs before release. Common fields:

- `Plugin Name`: Human-readable name shown in WP Admin.
- `Description`: Short summary of what the plugin does.
- `Version`: Release version; keep in sync with readme and assets.
- `Requires at least`: Minimum WordPress version.
- `Requires PHP`: Minimum PHP version.
- `Author`: Author or organization name.
- `License`: GPL-compatible license for WordPress.org distribution.
- `Text Domain`: Translation domain, normally the plugin slug.
- `Domain Path`: Translation file path, commonly `/languages`.
- `Update URI`: Custom update identifier for plugins not updated through WordPress.org.
- `Requires Plugins`: Comma-separated dependency slugs when the plugin depends on other plugins.

Bad:

```php
/**
 * Plugin Name: Tools
 */
```

Good:

```php
/**
 * Plugin Name: Example Tools
 * Description: Adds editorial workflow helpers.
 * Version: 1.0.0
 * Requires at least: 6.5
 * Requires PHP: 8.1
 * Author: Example Org
 * License: GPL-2.0-or-later
 * Text Domain: example-tools
 * Domain Path: /languages
 */
```

## Activation, Deactivation, Uninstall

Use lifecycle hooks for one-time setup and teardown:

- `register_activation_hook()`: create initial options, schedule events, set default data, and flush rewrites after registering rewrite-dependent objects.
- `register_deactivation_hook()`: clear scheduled events, temporary state, and flush rewrites if needed.
- `uninstall.php`: preferred for clear, isolated uninstall cleanup when deleting plugin-owned data.
- `register_uninstall_hook()`: acceptable for simple static callback cleanup.

Do not do this on every request:

- Flush rewrite rules.
- Create or migrate database tables.
- Run remote HTTP calls.
- Scan large directories.
- Recalculate expensive caches.
- Start sessions or write options without a user action.

Bad:

```php
add_action( 'init', 'flush_rewrite_rules' );
```

Good:

```php
register_activation_hook(
	__FILE__,
	static function (): void {
		Example\Post_Types::register();
		flush_rewrite_rules();
	}
);
```

## Dependency Boundaries

- Use WordPress APIs first: Options API, Metadata API, HTTP API, Filesystem API, Cron API, REST API, Scripts API, and Settings API.
- Avoid global pollution. Prefix all unavoidable globals and keep bootstrap constants minimal.
- Avoid singletons as the default. Prefer explicit service construction and dependency injection; use singleton/factory access only when it simplifies WordPress hook integration and remains testable.
- Do not write directly to database tables when a WordPress API exists.
- Keep admin, REST, block, frontend, storage, and domain logic in separate services.
- Public hooks are part of the plugin API. Prefix and document them.

Bad:

```php
global $wpdb;
$wpdb->query( "UPDATE {$wpdb->postmeta} SET meta_value = '$value'" );
```

Good:

```php
update_post_meta( $post_id, '_example_value', sanitize_text_field( $value ) );
```

## Data Model Choices

- `options`: Site-level configuration. Use for settings and small durable values. Autoload only when needed on most requests.
- `post meta`: Data attached to posts, pages, CPT entries, or block-related entities. Register meta when exposing to REST.
- `custom post types`: Content-like records that need editorial UI, permissions, queries, revisions, REST exposure, or taxonomies.
- `custom tables`: High-volume, relational, log/event, reporting, or write-heavy data that does not fit posts/meta. Requires schema versioning, migrations, and uninstall policy.
- `transients`: Temporary cache with expiration. Never use as the only durable store.

Decision shortcut:

- Configuration: options.
- Per-content attribute: post meta.
- User-manageable content entity: CPT.
- Large operational dataset: custom table.
- Cache: transient or object cache.

## Frontend Asset Strategy

- Register/enqueue with `wp_enqueue_script()` and `wp_enqueue_style()` on the narrowest appropriate hook.
- Use stable, prefixed handles such as `example-tools-frontend`.
- Declare dependencies instead of assuming load order.
- Use plugin version or generated asset metadata for versioning.
- Load admin assets only on plugin screens.
- Load frontend assets only when needed: shortcode render, block presence, template context, or route-specific logic.
- For block builds, prefer `block.json` asset fields and generated `.asset.php` files where available.
- Use script modules only when current WordPress and build tooling support the required module workflow.

Bad:

```php
add_action( 'wp_enqueue_scripts', static function () {
	wp_enqueue_script( 'app', plugins_url( 'app.js', __FILE__ ) );
} );
```

Good:

```php
add_action(
	'wp_enqueue_scripts',
	static function (): void {
		wp_enqueue_script(
			'example-tools-frontend',
			plugins_url( 'build/frontend.js', EXAMPLE_TOOLS_FILE ),
			array( 'wp-element' ),
			EXAMPLE_TOOLS_VERSION,
			true
		);
	}
);
```

## Admin Feature Strategy

- Gate every admin screen with the matching capability.
- Protect state-changing actions with nonces and capability checks.
- Prefer Settings API for settings pages that store options.
- Use admin-post handlers or REST endpoints for form actions; avoid processing writes during page render without explicit action checks.
- Escape values in fields, notices, and table output.
- Keep admin-only services registered from admin hooks or behind `is_admin()` when practical.

Bad:

```php
if ( isset( $_POST['example_option'] ) ) {
	update_option( 'example_option', $_POST['example_option'] );
}
```

Good:

```php
if (
	current_user_can( 'manage_options' ) &&
	check_admin_referer( 'example_save_settings' )
) {
	update_option(
		'example_option',
		sanitize_text_field( wp_unslash( $_POST['example_option'] ?? '' ) )
	);
}
```

## Agent Checklist Before Writing Plugin Code

- Identify plugin size: single file, small modular, or full modular architecture.
- Inspect existing structure and follow local conventions unless they are unsafe.
- Confirm header fields and minimum WordPress/PHP versions.
- Define ownership boundaries: bootstrap, admin, REST, blocks, storage, frontend, privacy, uninstall.
- Choose data storage deliberately: option, meta, CPT, custom table, transient.
- Plan lifecycle hooks and avoid per-request setup work.
- Decide whether Composer autoload is needed.
- Prefix or namespace every public symbol and persistent key.
- Use WordPress APIs before direct SQL, filesystem access, HTTP, or custom routing.
- Plan capabilities, nonces, sanitization, validation, and escaping before coding.
- Decide asset loading scope and handles before enqueueing scripts/styles.
- For blocks, prefer `block.json` and server-side registration.
- Add tests/checks proportional to risk and state any unavailable tooling.
